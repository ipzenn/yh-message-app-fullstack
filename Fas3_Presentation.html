Samtliga endpoints i server.js saknar hastighetsbegränsning. CodeQL flaggar raderna 28, 84, 141, 154, 164 och 187 — inklusive DELETE /messages/:id som AI-granskningen markerar som kritisk av en separat anledning.

Tillåtande CORS-konfiguration
OWASP A05
Vad CodeQL hittade

server.js:20 använder origin: "*" vilket tillåter alla domäner att göra cross-origin-anrop till API:et. CodeQL identifierar detta som en explicit misconfiguration.

// server.js:19 — flaggad av CodeQL app.use(cors({ origin: "*" })) // Åtgärd — begränsa till känd frontend app.use(cors({ origin: process.env.FRONTEND_URL }))
Notering

I kombination med den oskyddade DELETE-routen möjliggör detta att en extern webbplats triggar radering av meddelanden från en besökares webbläsare.

AI-kodgranskning
S1 & S3 — Kritiska fynd
S1 — Critical
Unauthenticated DELETE
OWASP A01
Problemet

DELETE /messages/:id saknar authenticateUser-middleware. Eftersom GET /messages returnerar alla message-ID:n publikt kan en angripare loopa igenom dem och radera hela meddelandeboardet utan att logga in.

// PATCH — korrekt skyddad app.patch("/messages/:id", authenticateUser, handler) // DELETE — ingen middleware alls app.delete("/messages/:id", /* tom */, handler)
Åtgärd
app.delete("/messages/:id", authenticateUser, async (req, res) => { if (message.user.toString() !== req.user._id.toString()) return res.status(403).json({...}) await message.deleteOne() })

JWT i console.log
OWASP A09

05 / 07
Riskanalys
Kombinationsrisker

Enskilda brister är allvarliga — men det verkligt farliga är hur de förstärker varandra. Tre kombinationer sticker ut:

🔴 Attack 1 — Massradering utan inloggning
S1 (oskyddad DELETE) + S7 (CORS wildcard) + CodeQL rate limiting: En extern webbplats kan trigga DELETE-anrop mot alla kända message-ID:n från en besökares webbläsare. Ingen inloggning krävs. Ingen begränsning på antal anrop. Hela databasen kan tömmas på sekunder.
🔴 Attack 2 — Credential stuffing i full hastighet
S4 (user enumeration via distinkta felmeddelanden) + U1/CodeQL (ingen rate limiting på /login): Angripare kan först identifiera alla existerande konton via felmeddelanden, och sedan köra lösenordsgissning mot de validerade kontona utan hastighetsbegränsning.
🟠 Attack 3 — Dubbel exponering av JWT
S3 (token i console.log) + Dependabot jsonwebtoken (signaturvalideringsbypass): JWT:s läcker aktivt till konsolen OCH det underliggande biblioteket har kända sårbarheter som kan möjliggöra att förfalskade tokens accepteras. Autentiseringen är komprometterad i båda riktningarna.
