KR-01 & KR-04
KR-01
Lösenordsstyrka
❌ Uppfylls ej
Problemet i koden

I /register-routen skickas lösenordet direkt till bcrypt.hash() utan någon längdvalidering. User-schemat i MongoDB saknar också min/max på lösenordsfältet.

// server.js — ingen validering sker const hashedPassword = await bcrypt.hash(password, 10) // ← 1 tecken godkänns lika gärna som 200
Åtgärd

Lägg till en explicit kontroll innan hashen skapas.

if (!password || password.length < 8 || password.length > 50) { return res.status(400).json({ message: "Password must be 8–50 chars" }) }
KR-04
Felmeddelanden vid inloggning
❌ Uppfylls ej
Problemet i koden

/login-routen returnerar två distinkta felmeddelanden beroende på vad som gick fel. En angripare kan därmed ta reda på om ett konto existerar — s.k. user enumeration.

// Avslöjar att kontot inte finns: message: "No account found with that username or email" // Avslöjar att kontot finns men fel lösenord: message: "Password is incorrect"
Åtgärd

Ersätt båda med ett identiskt, generiskt svar.

return res.status(401).json({ message: "Invalid credentials" }) // Används i båda felfall — inget avslöjas

Kodanalys
KR-02 — Åtkomstkontroll
⚠ Delvis uppfyllt — kritisk lucka

Middleware authenticateUser verifierar JWT-token och används korrekt på POST och PATCH. Men DELETE-routen saknar den helt — vem som helst kan radera ett meddelande utan att vara inloggad.

Skyddade routes ✓
app.post("/messages", authenticateUser, handler) app.patch("/messages/:id", authenticateUser, handler)
Oskyddad route ✗
app.delete("/messages/:id", /* ingen middleware */ async (req, res) => { // Ingen koll på vem som anropar await message.deleteOne() })
Åtgärd — lägg till authenticateUser + ägarskontroll
app.delete("/messages/:id", authenticateUser, async (req, res) => { const message = await Message.findById(req.params.id) if (message.user.toString() !== req.user._id.toString()) { return res.status(403).json({ error: "You can only delete your own messages" }) } await message.deleteOne() })

KR-03 — Brute-force-skydd
❌ Saknas helt

/login-routen har inga begränsningar på antalet inloggningsförsök. En angripare kan skicka tusentals lösenordsförsök per sekund från samma IP utan att blockeras.

Nuläget — obegränsade försök
app.post("/login", async (req, res) => { // Inga begränsningar // 1 000 försök/sekund? Inga problem. const user = await User.findOne(...) ... })
Åtgärd — express-rate-limit
import rateLimit from "express-rate-limit" const loginLimiter = rateLimit({ windowMs: 15 * 60 * 1000, // 15 min max: 10, // max 10 försök/IP message: "Too many attempts" }) app.post("/login", loginLimiter, async (req, res) => { ... })
Om paketet express-rate-limit

Det är det de facto standardverktyget för rate limiting i Express och länkas i Express officiella säkerhetsguider. Paketet har 10+ miljoner nedladdningar per vecka på npm och är aktivt underhållet. Trots det bör man alltid verifiera med npm audit och låsa versioner i package-lock.json.

⬆ Högst prioritet
